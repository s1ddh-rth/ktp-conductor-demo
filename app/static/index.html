What this is

UK Distribution Network Operators (DNOs) are under regulatory and operational pressure to digitise their low-voltage networks. Many LV cable runs were never mapped, or exist only on paper records — a problem at the heart of supporting EVs, heat pumps, and rooftop solar. The prototype is the kind of system a Knowledge Transfer Partnership (KTP) between a university and a DNO would build out, decomposed into the four components below. The first three (segmentation+vectorisation, hidden-cable inference, LiDAR classification) each have their own tab; the fourth (RGB+LiDAR fusion) is reached by ticking Run with RGB + LiDAR fusion on Tab 1, which calls the /api/fuse endpoint. Tab 4 is documentation, not a fifth component.

Tab 1 · Segment & Vectorise
A trained U-Net (ResNet-34 encoder) turns an aerial RGB image into a binary cable mask, a skeleton walker turns the mask into a graph, and the graph is simplified to GIS-ready GeoJSON LineStrings. Single learned component in the prototype.
Tab 2 · Infer Hidden Cables
Two deterministic baselines for cables not directly visible: a closed-form catenary fit for occluded spans, and a Steiner-tree approximation for predicting the cable network connecting buildings to a transformer. Performance floor a learned successor (GraphSAGE / GAT) would beat in the full project.
Tab 3 · LiDAR Classification
Per-point eigenvalue features (Demantké et al., 2011) classify points into ground / vegetation / conductor / structure. Hand-crafted features as a published baseline; PointNet++ named as Phase 2 in the roadmap.
RGB + LiDAR Fusion (/api/fuse)
A late-fusion endpoint re-scores each predicted LineString with LiDAR-derived linearity and conductor-fraction support. Decision-level baseline; mid-fusion (BEVFusion / TransFusion) named as Phase 2.

Trained on the TTPLA dataset (transmission-line aerial imagery) as a proxy for unavailable LV-specific data — the methodology doc is explicit about the domain gap and what would change for real LV imagery.

In operational terms

Picture a UK residential street where the low-voltage cable run from the local substation to each house was last surveyed on paper decades ago. A field crew currently maps it by eye from the road, often by tracing what's visible on overhead poles. This prototype reads an aerial image of that street and emits the cable run as a GIS-ready vector — a starting point a human engineer then verifies and approves before any record is updated. It is decision support, not an authoritative source: every output of this kind would be reconciled against the data-governance framework (ESQCR 2002 for electrical safety, IEC 61968 for utility data interchange, NUAR for underground asset registration, GDPR / ISO 27001 for the security envelope) before feeding any operational system.

For the deployment story end-to-end — Docker, systemd, healthcheck, Cloudflare tunnel, ArcGIS integration plan in Phase 3 of the roadmap — see the research roadmap and risk register.

What this tab demonstrates. The full RGB → vector pipeline. Drop in an aerial photo (or pick an example) and the server runs four stages:
  1. Segment. A U-Net with a ResNet-34 encoder, fine-tuned on TTPLA, runs sliding-window inference at 768 × 768 with Hann-windowed stitching for arbitrary input sizes. Output: a per-pixel "cable / not-cable" probability map.
  2. Threshold & clean. Binarise the probabilities (default τ = 0.5; the slider exposes this), then morphological-open to drop sub-cable noise. Output: the red mask overlay you'll see on the canvas.
  3. Skeleton → graph. Reduce the mask to a 1-pixel-wide medial axis, walk it to find junctions and endpoints, and simplify each chain with Douglas-Peucker. Output: a NetworkX graph with simplified Shapely LineStrings.
  4. GeoJSON. Project pixel coordinates onto a synthetic lat/lon box near Thatcham (purely a visualisation choice — the imagery isn't georeferenced) and emit a FeatureCollection. Output: the lines you see on the Leaflet map and that the Download GeoJSON button serialises.

Adjust the confidence threshold slider after a successful run — the pipeline re-runs at the new threshold without re-uploading the image. Tick Run with RGB + LiDAR fusion to call /api/fuse instead, which re-scores each predicted LineString against the curated LiDAR sample and reports a per-feature fused_confidence (visible as map opacity and in the tooltip on each line). For synthetic-LV examples, tick Show ground truth to overlay the procedurally-generated mask in green.

What this tab demonstrates. Two techniques for inferring cables that aren't directly visible in imagery — run on canonical pre-defined geometry, not on the synthetic LV examples from Tab 1. Pick a scenario below and click Run inference.
  • Hidden span: two known pole tops with the cable between them occluded → fit a physical catenary curve.
  • Network completion (no evidence): a transformer plus building footprints → predict the minimum-cost cable network that connects them all (Steiner-tree approximation).
  • Network completion (partial evidence): same scene with some cables already visible — the predicted tree is biased to align with those fragments.

Method

Hidden span (catenary): two black dots are the visible pole tops 60 m apart. The cable between them is occluded. We fit a physical catenary curve (gravity-driven sag) constrained only by the two anchors; the thin band shows sag uncertainty. With short spans the curve looks nearly straight — that is physically correct, not a bug.

Deterministic geometric classifier establishing the performance floor for LiDAR classification. Phase-2 successor: learned point-cloud network (PointNet++ / MinkowskiNet).

What this tab demonstrates. Per-point LiDAR classification using eigenvalue features (Demantké et al., 2011) — a published classical baseline, no deep learning. The local k-NN covariance matrix tells us whether a point's neighbourhood is shaped like a line (cable / pole), a plane (ground / roof), or a blob (vegetation). Combined with height-above-ground, four classes drop out cleanly.
  • What you should see: a flat slate-grey ground plane, two green vegetation blobs off to the sides, three vertical red poles down the middle, and a thin amber band of cable points strung between them at ~9 m height.
  • How to look: click Load & classify, then drag in the canvas to orbit. The cable band is easiest to spot from a slight side-angle — straight overhead it disappears into the structure points.
  • Why this is here: the JD asks for "RGB and LiDAR analysis with feature extraction and fusion". This baseline classifier feeds the /api/fuse endpoint on Tab 1; PointNet++ / MinkowskiNet are named as the Phase-2 learned successors in the roadmap.

Methodology summary

This prototype was built as a five-day sprint exploring an LV-conductor mapping pipeline. It is intentionally narrow: a working spine through four problem stages — detection, vectorisation, hidden-cable inference, and LiDAR classification — rather than a research contribution.

Stack

  • Segmentation: U-Net (ResNet34 encoder, ImageNet-pretrained) trained with focal + Dice loss on TTPLA at 768×768. PyTorch + segmentation-models-pytorch.
  • Vectorisation: scikit-image skeletonize → custom graph extraction → Shapely Douglas-Peucker simplification → GeoJSON.
  • Hidden-cable inference: catenary curve fit (least-squares, SciPy) for two-anchor cases; Steiner-tree approximation (NetworkX) over building footprints for topology completion.
  • LiDAR classification: per-point covariance eigenvalues (linearity / planarity / sphericity / verticality, Demantké et al. 2011) + height-above-ground thresholds.
  • Serving: FastAPI / uvicorn on a Fedora laptop (GTX 1050 Ti for inference); cloudflared named tunnel for HTTPS exposure.

What this prototype does not do

  • Real LV imagery — TTPLA is rural transmission lines. A full project would need an active labelling campaign on a DNO partner's own urban aerial / drone imagery, plus self-supervised pretraining on their unlabelled archive.
  • Real geo-referencing — pixel coordinates are mapped onto a synthetic lat/lon box for visualisation. Production work would consume camera-pose metadata from the survey provider.
  • Mid-level (learned) RGB+LiDAR fusion — the prototype implements a decision-level late-fusion baseline (the /api/fuse endpoint, callable from Tab 1). A learned mid-fusion model with cross-attention between modality-specific encoders (BEVFusion / TransFusion family) is named as Phase-2 in the research roadmap.
  • A graph neural network for topology completion — the demo uses a deterministic Steiner-tree approximation. A learned GNN conditioned on building footprints, transformer locations, and observed fragments is the obvious research direction.
